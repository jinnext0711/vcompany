```markdown
# ================================================================
# PHASE 1: ストーリー構造（Storyline Architecture）
# ================================================================

## 1-1. ストーリータイプの選定

以下の5つのストーリー構造から、
資料の種類・対象者・目的に最も適合するものを1つ選択し、
選択理由を述べること。

### Type A: SCQA（Situation → Complication → Question → Answer）
- 用途: 戦略提案、問題解決型プレゼン、コンサル提案書
- 特徴: 聴衆が「既知の状況」から入り、「予想外の複雑さ」に気づき、
  「本質的な問い」が生まれ、「解決策」に自然に導かれる
- McKinsey、BCGの標準フォーマット

### Type B: Problem → Insight → Solution → Proof → Ask
- 用途: ピッチデッキ、資金調達、コンペ
- 特徴: 問題の「大きさ」で掴み、非自明な洞察で差別化し、
  解決策の「なぜ我々か」を証明し、具体的な依頼で締める
- Y Combinator、500 Global等のDemo Day標準

### Type C: Why Now → What → How → Evidence → Vision
- 用途: 新規事業提案、タイミングが重要な提案
- 特徴: 「なぜ今このタイミングなのか」から入ることで緊急性を創出
- a16z（Andreessen Horowitz）が好むフォーマット

### Type D: Current State → Gap Analysis → Target State → Roadmap
- 用途: 社内改革提案、DX推進、業務改善
- 特徴: 現状を定量的に示し、あるべき姿とのギャップを可視化し、
  到達するためのロードマップを提示
- 社内コンサルティング、PMO（Project Management Office）の標準

### Type E: Executive Summary → Deep Dive → Appendix
- 用途: 読み資料（プレゼンなし）、取締役会資料、DD（Due Diligence）資料
- 特徴: 最初の3ページで結論と根拠の全体像を示し、
  以降で詳細を展開。読み手が「どこまで読むか」を選べる構造
- Amazon 6-pager の思想を応用

## 1-2. ストーリーの骨格設計

選択したストーリータイプに基づき、以下を定義すること:

### 1文サマリー（Elevator Pitch）
- 30秒で資料全体の主張を伝える1文
- 構造: [誰の] [何の問題を] [どう解決し] [どんな成果を生む]
- この1文がストーリー全体の「背骨」になる

### キーメッセージ（3つ以内）
- 聴衆が資料を見た後に「覚えていてほしいこと」を3つ以内で
- 各メッセージは1文、15字以内が理想
- 抽象的な理念ではなく、具体的で検証可能な主張であること

### ストーリーアーク（感情曲線）
- 資料を通じて聴衆の感情がどう推移するかを設計
- 例（Type B の場合）:
  1. 問題提起 → 「これは深刻だ」（危機感）
  2. インサイト → 「そうだったのか」（驚き・納得）
  3. 解決策 → 「これなら解決できるかもしれない」（期待）
  4. 証拠 → 「実際にうまくいっている」（信頼）
  5. 依頼 → 「一緒にやりたい」（行動意欲）

# ================================================================
# PHASE 2: 問題設定の精度（Problem Framing）
# ================================================================

問題設定は資料の「勝敗の8割」を決めます。
ここが甘いと、どれだけビジュアルが美しくても説得力がゼロになります。

## 2-1. 問題の階層構造

以下の3層で問題を定義すること:

### Layer 1: 表層の問題（Observable Problem）
- 聴衆が「知っている」「同意できる」レベルの問題
- データで裏付けられる事実ベースの記述
- 例: 「中小企業の70%がDXに着手できていない」

### Layer 2: 構造的原因（Structural Root Cause）
- なぜ表層の問題が解決されていないのかの構造的分析
- Issue Tree（論点ツリー）で分解
- MECE（Mutually Exclusive, Collectively Exhaustive: 相互排他・網羅的）に分解されていること
- 例: 「着手できない原因は ①人材不在（62%） ②コスト不透明（24%） ③経営層の理解不足（14%）」

### Layer 3: 本質的課題（Core Insight / Reframing）
- 従来の問題認識を覆す、より深い洞察
- 「実は問題はXではなくYだった」というリフレーミング
- これが競合提案との差別化ポイントになる
- 例: 「問題はDXの技術ではなく、経営判断に必要な『データで語る文化』の不在である」

## 2-2. 問題の定量化

以下のフレームワークで問題の「大きさ」を定量化すること:

```
【問題の大きさ】= 影響人数 × 1人あたりの損失 × 頻度
例: 中小企業380万社 × 年間生産性損失300万円 × 1回/年 = 年間11.4兆円の機会損失

【問題の緊急性】を示す指標:
- 時間経過による悪化率（例: 毎年5%の市場シェア低下）
- 競合環境の変化速度（例: 海外勢が3年以内に参入予定）
- 規制・制度変更のデッドライン（例: 2026年のインボイス制度完全適用）

【問題の「自分ごと化」】:
- 聴衆自身に影響する形で表現
- 「御社の場合、年間○○円の機会損失に相当します」
- 「このまま何もしなければ、3年後には○○になります」
```

## 2-3. 問題設定の品質チェック

以下の全項目にYesと答えられなければ、問題設定をやり直すこと:

- [ ] 問題は「解決可能」か？（天気のように制御不能な問題ではないか）
- [ ] 問題は「検証可能」か？（データで裏付けられるか）
- [ ] 問題の「持ち主」は明確か？（誰の問題かが曖昧でないか）
- [ ] 問題は「十分に大きい」か？（解決する価値があるか）
- [ ] 問題は「十分に絞られている」か？（広すぎて解決策が曖昧にならないか）
- [ ] Layer 3のリフレーミングは「非自明」か？（聴衆が「なるほど」と思うか）

# ================================================================
# PHASE 3: 論理構造（Logic Structure）
# ================================================================

## 3-1. ピラミッド・ストラクチャー

Barbara MintoのPyramid Principleに基づき、
資料全体の論理構造をピラミッド型で設計すること。

```
        [主張（1文サマリー）]
       /         |          \
   [根拠1]    [根拠2]     [根拠3]
   / | \      / | \       / | \
 [証拠] [証拠] [証拠]  [証拠] [証拠]  [証拠]
```

ルール:
- トップの主張は1つ（1文サマリーと一致）
- 根拠は2〜4つ（5つ以上は整理不足の証拠）
- 各根拠はMECEであること（重複なし・漏れなし）
- 各根拠の下に最低2つの証拠（データ、事例、論理）
- 「So What?（だから何？）」テスト: 各証拠から根拠へ、根拠から主張へ、
  「だから何？」と問うて論理が繋がること
- 「Why So?（なぜそう言える？）」テスト: 主張から根拠へ、根拠から証拠へ、
  「なぜそう言える？」と問うて答えられること

## 3-2. スライドごとの「So What」設計

全スライドにはスライドタイトルとは別に
「このスライドのSo What（だから何が言いたいのか）」を定義すること。

```
【悪い例】
スライドタイトル: 「市場規模の推移」
So What: なし（データを見せているだけ）

【良い例】
スライドタイトル: 「国内DX市場は年率23%で成長し、2027年に3.2兆円に到達」
So What: 「この成長速度は、今参入しなければ2年後には競合に席を取られることを意味する」
```

ルール:
- スライドタイトルは「主張」であること（「市場規模」ではなく「市場規模は○○」）
- タイトルは1行、最大2行。読み上げて自然な文であること
- 各スライドのSo Whatが、ピラミッドの論理構造のどこに位置するかを明記

## 3-3. 論理の接続詞チェック

スライド間の論理的接続を以下の接続詞で検証すること:

- スライドN → N+1 が「したがって（Therefore）」で繋がるか
- スライドN → N+1 が「さらに（Moreover）」で繋がるか
- スライドN → N+1 が「一方で（However）」で繋がるか

3つのいずれにも当てはまらない場合、そのスライドの位置が間違っているか、
不要なスライドである可能性が高い。

# PHASE 4: インサイトの深さ（Insight Quality）
# ================================================================

## 4-1. インサイトの5段階評価

資料内のすべての主要な主張を以下の5段階で評価し、
Level 3以上のみを採用すること。

```
Level 1: 事実の列挙（Observation）
  「市場規模は1.5兆円である」
  → 誰でも調べれば分かる。価値ゼロ。

Level 2: トレンドの指摘（Trend）
  「市場規模は年率23%で成長している」
  → 方向性は分かるが、行動に繋がらない。

Level 3: 因果関係の発見（Causation）
  「リモートワーク普及がDX投資を加速させ、年率23%成長を牽引している」
  → なぜそうなっているかの構造が見える。最低合格ライン。

Level 4: 非自明な構造の発見（Non-obvious Structure）
  「DX投資の80%は大企業に集中しており、中小企業向け市場は
   実質的にブルーオーシャンだが、参入障壁は技術ではなく
   '意思決定支援'にある」
  → 既存の認識を覆し、具体的な行動方針が導ける。

Level 5: パラダイムの転換（Paradigm Shift）
  「DXの本質は技術導入ではなく、組織の意思決定プロセスの再設計であり、
   これを理解している企業だけが次の10年で生き残る」
  → 聴衆の世界観を変える。ピッチコンテスト入賞レベル。
```

## 4-2. インサイト生成のフレームワーク

以下の手法を使い、素材からLevel 3以上のインサイトを抽出すること:

### 逆説アプローチ（Counterintuitive Framing）
- 業界の「常識」を特定する
- その常識が成り立たないケースを探す
- 常識と現実のギャップがインサイトになる
- 例: 常識「DXにはIT人材が必要」→ 現実「IT人材がいてもDXが進まない企業がある」
  → インサイト「DX成功の鍵はIT人材ではなく、経営層のデータリテラシー」

### 異業種アナロジー（Cross-industry Analogy）
- 同じ構造の問題が別業界でどう解決されたかを探す
- 例: 「中小企業のDXは、2010年代の飲食店のクラウドPOS導入に構造が似ている」
  → 成功パターンを転用できる

### 数値の再解釈（Data Reframing）
- 同じ数値を別の切り口で表現する
- 例: 「DX未着手70%」→「年間11.4兆円の生産性が眠っている」
- 絶対値 ↔ 相対値、ストック ↔ フロー、全体 ↔ 1人あたり の変換を試す

## 4-3. インサイトの品質チェック

- [ ] 「それは面白い」と聴衆が言うか？（退屈でないか）
- [ ] 「それは本当か？」と聴衆が確認したくなるか？（刺激的か）
- [ ] 「だからどうすればいい？」の答えが明確か？（行動に繋がるか）
- [ ] 30秒で説明できるか？（複雑すぎないか）
- [ ] 競合のプレゼンでは出てこないか？（差別化されているか）

# PHASE 5: ビジュアルコミュニケーション設計（Visual Communication）
# ================================================================

## 5-1. スライドごとのビジュアルタイプ選定

各スライドの「伝えたい情報の種類」に基づき、
最適なビジュアルタイプを選定すること。

```
【定量比較】→ 横棒チャート or テーブル
  いつ: 3つ以上の項目を数値で比較
  例: 競合比較、セグメント別売上

【推移・トレンド】→ 折れ線チャート or エリアチャート
  いつ: 時間軸に沿った変化を示す
  例: 市場規模推移、KPI推移

【構成比】→ 積み上げ棒グラフ or ウォーターフォール
  いつ: 全体の中の割合を示す
  円グラフは原則禁止（正確な比較が困難なため）
  例外: 2〜3区分で圧倒的な差がある場合のみ許容

【プロセス・フロー】→ フローチャート or ステップ図
  いつ: 手順、工程、ビジネスモデルの流れ
  例: ユーザージャーニー、オペレーションフロー

【関係性・構造】→ アーキテクチャ図 or マトリクス
  いつ: 要素間の関係を示す
  例: エコシステム、2x2マトリクス

【Before/After】→ 2カラム対比
  いつ: 変化のインパクトを示す
  例: 導入前後の比較、現状とあるべき姿

【1つの数字で語る】→ Big Number + コンテキスト
  いつ: 最も重要な1つの数値を印象づける
  例: "ROI 200%"、"年間11.4兆円"
  数字は60pt以上、その下に1行のコンテキスト文

【引用・顧客の声】→ 引用カード
  いつ: 社会的証明、ユーザーの実感
  例: 「導入後、意思決定速度が3倍になった」— A社 CTO
```

## 5-2. 情報密度の管理

```
【1スライド1メッセージの法則】
- 各スライドが伝えるメッセージは1つだけ
- 2つ以上ある場合はスライドを分割する
- 例外: Executive Summaryスライドのみ複数メッセージ可

【5秒ルール】
- スライドを見て5秒以内に「何が言いたいか」が分かること
- 5秒で分からなければ情報が多すぎるか、階層が不明確

【テキスト量の上限】
- ピッチデッキ: 1スライドあたり最大30ワード（日本語は50字）
- 戦略レポート: 1スライドあたり最大60ワード（日本語は100字）
- 読み資料: 1スライドあたり最大120ワード（日本語は200字）

【データ・インク比（Data-Ink Ratio）の最大化】
- Edward Tufteの原則: インクの大部分はデータを表現するために使う
- 装飾的な要素（グラデーション、3D効果、影）を最小化
- グリッド線、軸ラベル、凡例は必要最小限に
```

## 5-3. スライドマスタープラン

全スライドの設計を以下のフォーマットで一覧化すること:

```
Slide #: [番号]
Title: [主張型タイトル]
So What: [このスライドが全体の論理で果たす役割]
Visual Type: [5-1で選定したタイプ]
Key Data: [このスライドの核となるデータポイント]
Layout: [2カラム / フル幅 / Big Number / チャート+テキスト 等]
Notes: [プレゼン時の補足、アニメーション指示等]
```

# PHASE 6: 投資・実行視点（Execution Credibility）
# ================================================================

※ ピッチデッキ、資金調達、事業提案の場合に適用。
  社内向けレポートの場合はこのフェーズをスキップ可能。

## 6-1. ビジネスモデルの構造化

以下を明確に定義すること:

```
【収益モデル】
- 課金形態: サブスクリプション / 従量課金 / ライセンス / フリーミアム
- 単価: ○○円/月 or ○○円/取引
- LTV（Life Time Value: 顧客生涯価値）: ○○円
- CAC（Customer Acquisition Cost: 顧客獲得コスト）: ○○円
- LTV/CAC比: ○○倍（3倍以上が健全の目安）
- ユニットエコノミクスの成立時期: ○○ヶ月目

【市場規模の3層構造】
- TAM（Total Addressable Market: 獲得可能な最大市場規模）: ○○兆円
- SAM（Serviceable Addressable Market: 実際にアプローチ可能な市場規模）: ○○億円
- SOM（Serviceable Obtainable Market: 短期的に獲得可能な市場規模）: ○○億円
- TAM → SAM → SOM の絞り込みロジックを明記すること
  （「日本の中小企業380万社のうち、従業員50人以上の製造業12万社がSAM」等）

【成長ドライバー】
- 1年目: ○○（例: 創業者の人脈による直販）
- 2年目: ○○（例: パートナー経由の間接販売開始）
- 3年目: ○○（例: プロダクト主導の自律的成長）
```

## 6-2. トラクション（実績）の見せ方

```
【数値実績がある場合】
- MoM（Month over Month: 月次成長率）を折れ線で表示
- コホート分析（月別の継続率推移）があれば追加
- 「右肩上がり」のグラフは必須（投資家の最重要判断材料）

【数値実績が少ない場合】
- 代替指標を使う:
  - ウェイトリストの登録数
  - LoI（Letter of Intent: 意向表明書）の獲得数
  - パイロット顧客の声（定性 + 定量）
  - 技術的マイルストーンの達成状況
- 「早期だが勢いがある」という印象を作る

【実績ゼロの場合】
- ファウンダーの関連実績（「同領域で○年の経験」）
- 類似事例のベンチマーク（「類似サービスXは○年でARR○億円」）
- 初期仮説の検証結果（インタビュー、アンケート等）
```

## 6-3. Ask（依頼）の設計

```
【資金調達の場合】
- 調達額: ○○円
- 用途の内訳を「3つ以内」に（例: プロダクト開発60%、営業体制構築30%、運転資金10%）
- 到達目標（この資金で何を達成するか）を明記
- バリュエーションに言及するかは相手に応じて判断

【社内提案の場合】
- 必要な予算: ○○円
- 必要な人員: ○名（既存 + 新規）
- 必要な期間: ○ヶ月
