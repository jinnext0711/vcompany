# Claude Codeで高品質ビジュアルを作るための実践ガイド

## はじめに：なぜ品質が低くなるのか

Claude Codeが生成するビジュアルの品質が低い根本原因は、「LLM（Large Language Model）に直接デザインさせている」ことにあります。LLMはピクセル単位の美的判断が苦手で、毎回ゼロからコードを書くと一貫性も出ません。

解決策の核心は「分業」です。デザインの骨格（テンプレート・デザインシステム）は人間が決め、Claude Codeにはコンテンツの差し込みとレンダリングパイプラインの実行を任せる、という構造にすることで品質が劇的に改善します。

---

## 共通原則：品質を左右する3つのレバー

### 1. デザインシステムを先に定義する

Claude Codeに「かっこいい画像を作って」と言うのではなく、以下を事前に決めてプロンプトに含めます。

- カラーパレット（Primary / Secondary / Accent の3色 + 背景色）
- フォント指定（見出し用 / 本文用、日本語は Noto Sans JP 推奨）
- レイアウトのグリッド構造（余白、要素間のスペーシング）

```
# 例：プロンプトに含めるデザイントークン
- Primary: #1E2761 (Navy)
- Secondary: #CADCFC (Ice Blue)  
- Accent: #F96167 (Coral)
- 見出しフォント: "Noto Sans JP Bold", 32px
- 本文フォント: "Noto Sans JP Regular", 16px
- 余白: 上下左右 40px、要素間 24px
```

### 2. レンダリングパイプラインを固定する

生成方法によって最終品質が大きく変わります。用途別の推奨パイプラインは以下の通りです。

| 用途 | 推奨パイプライン | 品質 |
|------|-----------------|------|
| スライド | PptxGenJS → .pptx | ★★★★ |
| Note記事アイキャッチ | HTML/CSS → Canvas API → .png | ★★★★★ |
| 技術図解 | SVG直接生成 or Mermaid → .svg/.png | ★★★☆ |
| ポスター・アート系 | Canvas (node-canvas) → .png/.pdf | ★★★★★ |

### 3. QA（品質検証）を自動化する

生成 → 画像化 → 目視確認のサイクルを組み込むことが不可欠です。「一発で完成」ではなく「生成→検証→修正→再検証」を前提にします。

## 用途別：具体的なワークフロー

---

### A. スライド（プレゼン資料）

#### 推奨アプローチ：PptxGenJS

Claude Codeのスキルセットに含まれるPptxGenJSを使い、Node.jsスクリプトで.pptxを直接生成する方法が最も安定します。

#### プロンプト設計のコツ

```
以下の条件でPptxGenJSを使ったNode.jsスクリプトを書いてください。

# デザイン方針
- カラーパレット: Primary #1E2761, Secondary #CADCFC, Accent #FFFFFF
- 見出し: Georgia Bold 36pt
- 本文: Calibri 16pt
- タイトルスライドは暗い背景、コンテンツスライドは明るい背景
- 各スライドにビジュアル要素（図形・アイコン・チャート）を必ず含める
- テキストのみのスライドは禁止

# レイアウトルール
- 余白: 上下左右 0.5inch以上
- 要素間: 0.3inch以上
- タイトルは左揃え、36pt以上
- 同じレイアウトの連続は2スライドまで

# コンテンツ
（ここにスライドの内容を記載）
```

#### 品質向上のポイント

- テンプレート.pptxをFigmaやPowerPointで先に作り、python-pptxで内容だけ差し替える方式も有効
- 生成後は必ず画像化して目視確認する（LibreOffice → PDF → pdftoppm → JPG）
- タイトル下のアクセントライン（AI生成の典型的癖）は使わない
- 各スライドのレイアウトを変化させる（2カラム、グリッド、大きな数字、タイムラインなど）

### B. Note記事のアイキャッチ画像

#### 推奨アプローチ：HTML/CSS → node-canvas / Playwright

アイキャッチ画像は「1枚の画像ファイル」として出力する必要があるため、node-canvasで直接描画するか、HTMLをPlaywrightでスクリーンショットする方法が適しています。

#### パイプライン1：node-canvas（推奨）

```javascript
// Claude Codeに指示するプロンプト例
// node-canvasを使って1200x630pxのOGP画像を生成するスクリプトを書いてください
// 
// デザイン仕様：
// - サイズ: 1200x630px（noteのアイキャッチ推奨サイズ）
// - 背景: グラデーション (#1E2761 → #065A82)
// - タイトルテキスト: 白、Noto Sans JP Bold、40px、中央配置
// - サブタイトル: #CADCFC、Noto Sans JP Regular、20px
// - 装飾: 右下に幾何学パターン（丸や三角の組み合わせ）
// - フォントファイル: ./fonts/NotoSansJP-Bold.ttf を registerFont で登録
```

#### パイプライン2：HTML → スクリーンショット

HTMLでデザインを組み、Playwrightでキャプチャする方法は、CSSの表現力を活かせるため複雑なレイアウトに向いています。

```javascript
// Playwrightでのキャプチャ例
const { chromium } = require('playwright');

const browser = await chromium.launch();
const page = await browser.newPage();
await page.setViewportSize({ width: 1200, height: 630 });
await page.setContent(htmlString);
await page.screenshot({ path: 'eyecatch.png' });
await browser.close();
```

#### 品質向上のポイント

- 日本語フォント（Noto Sans JP / Noto Serif JP）は.ttfファイルを明示的に登録する
- noteのアイキャッチは1280x670pxまたは1200x630pxを目安にする
- テキストの可読性を最優先：背景にオーバーレイ（半透明の黒）を入れる
- ブランドカラー（自身のnoteのテーマカラー）を統一して使い回す
- テンプレートパターンを3〜5種類用意しておき、記事テーマで切り替える

#### テンプレートパターン例

| パターン | 構成 | 向いている記事 |
|---------|------|--------------|
| グラデーション背景 + 中央タイトル | シンプルで汎用的 | 一般的な解説記事 |
| 左右分割（色面 + テキスト） | モダンで目を引く | 技術記事、比較記事 |
| 大きな数字 + サブタイトル | インパクト重視 | ランキング、Tips系 |
| 背景パターン + オーバーレイ | 抽象的でプロ感 | 思考系、エッセイ系 |
| アイコン/図形 + タイトル | 情報を素早く伝える | How-to記事 |

### C. 技術的な図解・ダイアグラム

#### 推奨アプローチ：用途で使い分け

| 図の種類 | 推奨ツール |
|---------|-----------|
| フローチャート、シーケンス図 | Mermaid |
| アーキテクチャ図、ネットワーク図 | SVG直接生成 |
| データの可視化 | Chart.js → Canvas → PNG |
| ER図、クラス図 | Mermaid |

#### Mermaid活用法

Mermaidは記述量が少なく、Claude Codeとの相性が良いです。ただし見た目のカスタマイズには限界があるため、「構造を正確に伝える」用途に向いています。

```
# Claude Codeへの指示例
以下のシステムアーキテクチャをMermaidで書いてください。

# 要件
- graph TD（上から下）のフローチャート
- テーマ: dark
- ノードの色: Primary #1E2761, Secondary #065A82
- ユーザー → フロントエンド → API Gateway → マイクロサービス群
- マイクロサービスは3つ（認証、データ処理、通知）
- 各サービスからDBへの接続を表示

出力後、mmdc（mermaid-cli）でPNGに変換してください。
背景は白、幅1600px、DPI 2xで出力してください。
```

#### SVG直接生成（高品質な図解向け）

見た目の自由度を求めるなら、SVGを直接書かせる方法が有効です。

```
# Claude Codeへの指示例
以下のDX推進フローを示すSVG図を作成してください。

# デザイン仕様
- キャンバスサイズ: viewBox="0 0 1200 600"
- フォント: sans-serif（後でNoto Sans JPに置換可能）
- 色: 背景 #FAFAFA、ボックス #1E2761（角丸8px）、テキスト #FFFFFF
- 矢印: #065A82、太さ2px、先端は三角形
- ボックス間の間隔は均等に

# 内容
現状分析 → 課題特定 → ソリューション設計 → PoC実施 → 本番展開 → 効果測定
各ボックスの下に1行の説明テキスト（12px、#666666）

# 品質要件
- 要素が重ならないこと
- テキストがボックスからはみ出さないこと
- 全体のバランスが左右対称であること
```

#### 品質向上のポイント

- Mermaidは「構造の正確さ」が得意、SVGは「見た目の自由度」が得意
- 技術図解は「情報の正確さ > デザインの美しさ」が優先順位
- 色は3色以内に抑える（背景 + メイン + アクセント）
- フォントサイズは最小12px（読めないテキストは入れない）
- 生成後にSVGを開いて目視確認し、微調整を依頼する

## 横断的なプロンプトテクニック

### 1. 「デザイン哲学」を先に書かせる

canvas-designスキルのアプローチを応用し、いきなりコードを書かせず、まず「どういうデザイン方針で作るか」を.mdファイルに言語化させてから実装に入ると品質が上がります。

```
# ステップ1：デザイン方針を言語化
このDX解説記事のアイキャッチ画像のデザイン方針を書いてください。
トーン、色使い、構図、タイポグラフィの考え方を含めて。

# ステップ2：方針に基づいてコード生成
上記の方針に忠実に従い、node-canvasで画像を生成するスクリプトを書いてください。
```

### 2. 「やってはいけないこと」を明示する

Claude Codeは「やること」だけでなく「避けること」を伝えると精度が上がります。

```
# 避けること
- テキストのみの単調なレイアウト
- デフォルトフォント（Arial, sans-serif）の使用
- 紫グラデーション（AI生成の典型パターン）
- 要素の重なり・はみ出し
- 装飾なしの白背景
```

### 3. 二段階レビューを組み込む

```
# 指示の末尾に追加
生成後、以下を確認してください：
1. 画像をファイル出力して私に見せてください
2. 自分で画像を確認し、以下の問題がないか報告してください：
   - テキストの切れ・重なり
   - 色のコントラスト不足
   - レイアウトの偏り
   - フォントの未適用
3. 問題があれば修正した版も出力してください
```

## フォント管理のベストプラクティス

日本語の画像生成でフォントは最大の落とし穴です。

| 環境 | 推奨フォント | 入手方法 |
|------|------------|---------|
| node-canvas | Noto Sans JP | Google Fontsからダウンロードし registerFont() |
| Playwright/HTML | Noto Sans JP | HTMLの@font-face or Google Fonts CDN |
| PptxGenJS | 游ゴシック / メイリオ | OS標準搭載（閲覧環境に依存） |
| SVG | sans-serif → 後から置換 | SVGのfont-family属性を後で編集 |

node-canvasでのフォント登録：

```javascript
const { registerFont, createCanvas } = require('canvas');
registerFont('./fonts/NotoSansJP-Bold.ttf', { family: 'Noto Sans JP', weight: 'bold' });
registerFont('./fonts/NotoSansJP-Regular.ttf', { family: 'Noto Sans JP', weight: 'normal' });
```

## まとめ：品質を上げる最短ルート

1. テンプレートを用意する（毎回ゼロから作らせない）
2. デザイントークンをプロンプトに含める（色、フォント、余白）
3. 用途に合ったレンダリングパイプラインを選ぶ
4. 「やってはいけないこと」を明示する
5. 生成→検証→修正のサイクルを組み込む

この5つを押さえれば、Claude Codeの出力品質は大幅に改善します。
